# 🏛️ ПАСПОРТ ПРОЕКТА: Miyzamchi (Мыйзамчы)

**Miyzamchi** («Юрист» или «Законотворец») — это продвинутый искусственный интеллект и LegalTech-ассистент, специализирующийся на законодательстве Кыргызской Республики. Проект объединяет возможности LLM, векторных баз данных (Pinecone) и мощного WYSIWYG-редактора для интеллектуального анализа, редактирования и составления юридических документов. 

Этот документ является **глубоким реверс-инжинирингом** реальной кодовой базы (файлов `server.js`, `src/App.jsx`, `services/legalAgents.js`).

---

## 1. 📚 Инфраструктура RAG и Pinecone (Retrieval)

Логика RAG реализована в `services/legalAgents.js` с использованием каскадного векторного поиска и метаданных.

### Расширение запросов (Query Expansion) и Agent 1
Агент `expandQuery` (Extractor & Searcher) извлекает из фрагмента текста нормативную привязку.
Он получает `npa` (название НПА), `article` (номер статьи) и `queries` (набор переформулированных запросов).
**Системный промпт (EXPAND_SYS):**
```text
Ты — экстрактор и поисковик для базы законов Кыргызской Республики.
Из ОДНОГО фрагмента документа извлеки нормативную привязку и собери поисковые запросы.
Верни СТРОГО JSON:
{ "npa": "<полное название НПА или null>",
  "article": "<номер статьи из текста или null>",
  "queries": ["<запрос 1>", "<запрос 2>", "<запрос 3>"] }
ПРАВИЛА:
- npa: если упомянут нормативный акт ... верни его ПОЛНОЕ официальное название, раскрывая аббревиатуры (ГК КР → Гражданский кодекс Кыргызской Республики).
...
```

### Механика Pinecone Search и Фильтрация по Метаданным
В функции `pineconeSearch(queries, npa, topKPerQuery)` поиск идет по массиву сгенерированных запросов `queries`.
- **Жесткая привязка (`npa_title`):** Сначала берется точное имя базы данных (`getDbExactName(npa)`) и в Pinecone отправляется фильтр: `{ npa_title: { $eq: exact } }`. 
- **Soft-Fallback:** Если запрос с фильтром возвращает `0` результатов (что часто бывает при несовпадении названий или отсутствии статьи), система автоматически переключается на чистый семантический поиск без фильтра.
- **Иерархия контекста:** Найденные статьи делятся на `⭐ КЛЮЧЕВАЯ СТАТЬЯ` и `📚 ВСПОМОГАТЕЛЬНАЯ`, что помогает LLM отличать ядро нормы от процессуальных положений.

## 2. 🧠 Мультиагентная архитектура и Системные Промпты

Анализ документов построен на Map-Reduce пайплайне с участием нескольких агентов.

### Agent 2 (Checker / Validator)
Проверяет точность цитирования номеров статей и текстов.
**Системный промпт (VALIDATOR_SYS):**
```text
Ты — инспектор нормоконтроля по праву Кыргызской Республики.
Тебе дан ФРАГМЕНТ документа пользователя и ЭТАЛОННЫЙ текст закона из базы (RAG).
Сравни текст пользователя с эталоном. Твоя задача — проверить ТОЧНОСТЬ ЦИТИРОВАНИЯ...
Верни СТРОГО JSON:
{ "status": "correct" | "error" | "out_of_base",
  "marker": "✅ Верно" | "🔴 ОШИБКА" | "⚠️ Вне базы",
  "detail": "<если error: что не так и как исправить>",
  "cited_articles": ["<НПА, ст.N из эталона>"] }
ПРЕЗУМПЦИЯ НЕВИНОВНОСТИ ТЕКСТА (защита от ложных срабатываний — ВАЖНО):
- Если в тексте пользователя есть детали, которых ПРОСТО НЕТ в предоставленном куске эталона, НО они прямо ему НЕ противоречат — это НЕ ошибка.
```

### Agent 3 (Final Judge)
Модель рассуждения (обычно DeepSeek) синтезирует ответы, группирует ошибки по НПА и удаляет дубликаты. Вызывается функция `judge()`.
**Системный промпт (JUDGE_SYS):**
```text
Ты — Старший партнёр юридической фирмы (право Кыргызской Республики).
Тебе дан структурированный отчёт младших юристов — результаты нормоконтроля, СГРУППИРОВАННЫЕ ПО НПА.
Составь финальный отчёт для клиента... Убери ДУБЛИРУЮЩИЕСЯ ошибки...
```

## 3. 🎯 Интеллектуальная Маршрутизация и Классификация намерений

В `server.js` перед запуском тяжелого RAG происходит классификация намерений пользователя, чтобы понять, нужна ли *правовая проверка* или просто *техническая правка*.

### Kill-Switch (Обход RAG)
Если юрист просит исправить опечатку или заменить сумму, RAG не нужен. В системе зашит Regex-паттерн, который жестко перехватывает эти команды.
**Код из `server.js`:**
```javascript
const EDIT_COMMAND_REGEX = /(?:^|\s)(измени|измини|зиени|исправь|испарвь|замени|заемни|удали|удоли|поменяй|помеяй|установи|перепиши|сделай|подправь)(?!\p{L})/iu;
```
Если текст пользователя попадает под этот `EDIT_COMMAND_REGEX` (или его частые опечатки типа *зиени*, *испарвь*), режим принудительно переключается в **EDITOR**. 

### Двухступенчатая классификация намерений
Если Kill-Switch не сработал, запускается `classifyUserIntent(query)`:
1. **quickIntent:** Использует `editMarkers` Regex для технических правок и `legalMarkers` Regex для правовых вопросов (проверка соответствия, поиск рисков).
2. **llmIntent:** Если Regex не справился, вызывается легковесный вызов LLM. LLM классифицирует запрос на три категории:
   - `EDITOR` — техническая правка без последствий (изменить цифру). RAG обходится.
   - `RAG_AGENT` — правовая задача. Будет вызван `adaptiveRetrieval`.
   - `CLARIFY` — намерение размыто (например, "тут что-то не так с суммой"). Агент задает уточняющий вопрос.

## 4. 🛠️ Tool Calling и Split Execution

Miyzamchi реализует паттерн **Split Execution**: Бэкенд генерирует команды (JSON), а Фронтенд (React-приложение) применяет их к DOM через нативный движок редактора (`window.docEngine`).

### Схемы инструментов (Бэкенд)
В `server.js` (`AGENT_SYSTEM_PROMPT`) LLM проинструктирована возвращать массив команд для редактирования документа. Доступные инструменты (op):
```json
{
  "reasoning": "Кратко (1-2 предложения) что и почему меняешь...",
  "commands": [
    {"op": "replace", "old_text": "ТОЧНАЯ существующая фраза из документа", "new_text": "новый текст"},
    {"op": "insert_after", "anchor": "ТОЧНАЯ фраза-якорь из документа", "text": "новый абзац"},
    {"op": "insert_end", "text": "текст для добавления в конец"}
  ]
}
```

### Парсинг команд (Фронтенд - `App.jsx`)
На стороне клиента (в `src/App.jsx`) функция `parseAgentCommands(text)` ищет ````json` блок. Она поддерживает как формат массива `commands`, так и упрощенные инструменты `insertion_text` и `anchor_text`.

Она трансформирует op-команды массива `commands` во внутренние типы фронтенда:
- `replace` → `replace_smart` — точечная inline-замена `old_text` → `new_text`.
- `insert_after` / `insert_end` — вставка после якоря / в конец документа.
- `comment` → `comment` (Tier 2) — замечание на фрагмент (анкор + текст).
- `format` → `format` (Tier 2) — стиль фрагмента (анкор + marks).
- `replace_selection` — замена выделенного пользователем фрагмента.

### Выполнение (applyAgentCommand) — нативный SuperDoc API
Функция `applyAgentCommand(cmd, toastFn)` маршрутизирует команды в движок через нативный API (по `superdoc_docs.md`), все правки — в режиме Track Changes (`ensureTrackChanges`, автор «Мыйзамчи AI»):
1. **replace_smart** — `window.superdoc.search(old)` → `{from,to}` → `editor.commands.insertContentAt({from,to}, new)`. Фолбэк: поиск по text-нодам ProseMirror.
2. **insert_end / insert_after** — `editor.doc.insert({value, type:'markdown'})` либо вставка абзаца по якорю.
3. **comment** (Tier 2) — мост: поиск → `setTextSelection` → `editor.doc.selection.current().target` → `editor.doc.comments.create({target, text})`. Текст документа не меняется (аудит рисков).
4. **format** (Tier 2) — тот же мост → `editor.doc.format.apply({target, inline:marks})`. Фолбэк: `tr.addMark`.
5. Структурные ошибки → Toast («Фрагмент не найден») + `console.log` с `{from,to}`.

Контракт команд описан в ДВУХ промптах синхронно: `buildAgentPrompt` (фронт) и `AGENT_SYSTEM_PROMPT` (бэк). Главное правило: **запрос на правку → `replace`; запрос на анализ/поиск рисков → `comment`**.

## 5. 🚀 Деплой

- **Бэкенд** (`server.js` + routes) — Render, авто-деплой по push в `main`.
- **Фронтенд** (Vite/React/SuperDoc) — **СТРОГО ВРУЧНУЮ**: автодеплой/вебхуки Netlify отключены на стороне хостинга. Разработчик делает `npm run build` локально и перетаскивает папку `dist/` в дашборд Netlify (drag-and-drop). `netlify.toml` чинить для автосборки бесполезно. `dist/` в `.gitignore`.
