# Режимы Мыйзамчы — полная карта (расширенная)

Полная техническая карта всех режимов ИИ в проекте Мыйзамчы (`server.js`):
агенты, топ-K, пороги score, число вызовов, модели, параметры генерации, ретраи.

> Актуально на: 2026-05-22. Источник истины — `server.js`.
> Все числа взяты из констант кода, не приблизительные.

---

## 0. Краткая карта (TL;DR)

| Группа | Режим | Эндпоинт | Pinecone topK | LLM-вызовов* | Макс. агентов |
|---|---|---|---|---|---|
| **А** | Болталка | `/api/chat` `fast` | — (поиск выключен) | 1 | 1 |
| **А** | Быстрый | `/api/chat` `fast` | 10 | 2 | 1 |
| **А** | Думающий → Простой | `/api/chat` `thinking` | 15 | 2–3 | 2 |
| **А** | Думающий → Глубокий | `/api/chat` `thinking` | 5+10+8+6+5 = **34** | 7–8 | 7 |
| **Б** | Агент-редактор | `/api/chat` `agentMode` | 10 / 15 / 22 (адаптив) | 2 | 1 |
| **Б** | AI-редактор | `/api/edit` | — | 1 | 1 |
| **В** | Проверка документа | `/api/analyze-document` | 5 на статью/пункт | ~25–35 | **до 56** |
| **В** | Глубокий анализ (PRO) | `/api/deep-analyze-document` | 5 | ~12–20 | **17 LLM + до 30 проверок** |
| **Г** | Сравнение редакций | `/api/compare-documents` | — (cos-выравнивание) | ~104 (1 конт + 2 сегм + 100 ворк + 1 DS) | **до 104** |

\* «LLM-вызов» = запрос к генеративной модели. Эмбеддинги и Pinecone — отдельно.

## 1. Инфраструктура (общая для всех режимов)

### 1.1 Модели

| Назначение | Модель | Константа |
|---|---|---|
| Основная (PRIMARY) | `gemini-2.5-flash` (Google, stable) | `PRIMARY_MODEL` |
| Резервная (FALLBACK) | `gemini-2.5-flash-lite` (Google) | `FALLBACK_MODEL` |
| Эмбеддинги | `gemini-embedding-001`, **768 измерений** | `EMBEDDING_MODEL` |
| Судья / Старший партнёр | `deepseek-v4-pro` (DeepSeek) | `DEEPSEEK_MODEL` |

**DeepSeek V4 Pro:** OpenAI-совместимый SDK, `baseURL: https://api.deepseek.com`,
`reasoning_effort: 'high'`, `thinking: {type:'enabled'}`. Мягкая зависимость —
если ключа нет или DeepSeek упал, автоматический откат на Gemini.

### 1.2 RAG — профили `adaptiveRetrieval`

`adaptiveRetrieval(query, mode, res, opts)` шлёт в Pinecone `topK = maxK`,
затем фильтрует elbow-методом до `minK..maxK` статей.

| Профиль | maxK (= Pinecone topK) | minK (минимум после фильтра) |
|---|---|---|
| `fast` | 10 | 3 |
| `agent` | 15 | 4 |
| `thinking` | 25 | 5 |

> `opts.maxK` / `opts.minK` переопределяют дефолт профиля — многие режимы так и делают.

### 1.3 Пороги отбора статей (score)

| Параметр | Значение | Смысл |
|---|---|---|
| `absoluteMinScore` | **0.45** | ниже — статья отбрасывается полностью |
| `coreScoreThreshold` | **0.75** | ≥ 0.75 → ⭐ ключевая, ниже → 📚 вспомогательная |
| `elbowDropRatio` | **0.15** | обрыв ранжирования: падение score >15% → отсечка |
| `confidenceThresholds` | high 0.75 · medium 0.70 · low 0.60 | уверенность сверки статьи в анализе документа |

### 1.4 Параметры генерации

| Вызов | temperature | maxOutputTokens | стриминг |
|---|---|---|---|
| Болталка / Быстрый (`handleFast`) | по умолчанию (~1.0) | по умолчанию | да |
| Простой / Глубокий синтез (`streamGeminiResponse`) | по умолчанию | по умолчанию | да |
| Агент-редактор (`handleAgent`) | **0.4** · topP 0.9 | **4096** | да |
| AI-редактор (`/api/edit`) | по умолчанию | по умолчанию | нет |
| Судья — анализ документа (`runJudge`) | **0.2** | 4096 (при откате на Gemini) | да |
| Старший партнёр (`seniorPartnerSynthesis`) | **0.25** | 2048 (при откате) | нет |
| Драфтер (`drafterGenerate`) | **0.35** · topP 0.9 | **3500** | нет |
| Воркеры (`callOnce`) | по умолчанию | по умолчанию | нет |

### 1.5 Надёжность и ретраи

| Механизм | Где | Поведение |
|---|---|---|
| `generateContentResilient` | все воркеры через `callOnce` | **4 попытки** (maxRetries=3), backoff `800мс·2ⁿ` + jitter 0–800мс, потолок 8с. **Последняя попытка → FALLBACK_MODEL.** На 404 — сразу на резерв. |
| `handleFast` собственный ретрай | Болталка, Быстрый | до `KEYS.length` повторов, пауза 800мс, **модель не меняется** (всегда PRIMARY) |
| `handleAgent` собственный ретрай | Агент-редактор | до `min(KEYS.length, 3)`, пауза 1500мс |
| `deepseekCall` | Старший партнёр | **3 попытки**, backoff потолок 4с, затем откат на Gemini |
| `streamDeepSeekResponse` | Судья | 1 попытка; сбой до 1-го чанка → откат на Gemini-стрим |
| Блокировка ключа (`blockKey`) | ротация | **15 сек**, ТОЛЬКО при 429 (не при 503) |
| Ротация ключей | Gemini | round-robin по `GEMINI_API_KEY` (через запятую) |
| Rate limit `/api/chat` | — | **30 запросов / мин** |
| Rate limit `/api/analyze-document` | — | **30 запросов / мин** |
| Rate limit `/api/deep-analyze-document` | — | **6 запросов / мин** |
| Rate limit `/api/compare-documents` | — | **8 запросов / мин** |

---

## В.1 — Проверка документа (`/api/analyze-document`)

Оркестратор `analyzeDocumentSmart()`. Гибрид: проверка статей + проверка пунктов.

| Этап | Агент | Pinecone topK | LLM |
|---|---|---|---|
| 0. Паспорт документа | `extractDocumentContext` | — | 1 |
| 1. Извлечь статьи | `extractArticles` | — | 1 + по 1 на чанк (7000/500) |
| 1. Разбить на пункты | `segmentDocument` | — | 1 + по 1 на чанк (7500/400) |
| 2. Проверка статей | `verifySingleArticle` ×N | **5** на статью | 0 (regex-сверка) |
| 2. Проверка пунктов | `verifySegment` ×N | **5** на пункт | **1 на пункт** |
| 4. Заключение | `runJudge` | — | 1 DeepSeek V4 Pro |

**Лимиты-агенты:**

| Что | Макс. |
|---|---|
| Верификаторы статей (параллельно) | **30** (`maxArticles`) — без LLM |
| Верификаторы пунктов (параллельно) | **25** (`SEGMENT_LIMIT`) — каждый с LLM |
| Судья | 1 |
| **Итого макс. параллельных агентов** | **до 56** (30 + 25 + 1) |

**Пример** (12 статей + 25 пунктов, скриншот «30/41»):
~4–6 LLM (паспорт+извлечение+сегментация) + **25 LLM-вердиктов по пунктам** +
37 эмбеддингов + 37 Pinecone (topK 5) + 1 DeepSeek ≈ **30 LLM Gemini + 1 DeepSeek**.

## В.2 — Глубокий анализ PRO (`/api/deep-analyze-document`)

Оркестратор `runDeepAnalysis()`. Архитектура «Старший партнёр + команда».
**Параметры запроса:** `perspective` (`ours`/`opponent`/`audit`) ·
`modules` (любая комбинация `audit`, `strategy`, `drafter`, `mentor` — **макс. 4**).

| Агент / модуль | Под-агенты | Pinecone topK | LLM-вызовов |
|---|---|---|---|
| Паспорт + сегментация | — | — | 2 (+ чанки) |
| **Аудитор** (`audit`) | redFlags · collisions · procKiller · factCheck | factCheck: 5 на статью | 3 + 1 (экстрактор внутри factCheck) |
| └ factCheck → верификаторы статей | до 30 | 5 | 0 (regex) |
| **Стратег** (`strategy`) | heatmap + контраргументы | 5 на контраргумент | 1 + до **6** |
| **Драфтер** (`drafter`) | — | — | 1 |
| **Ментор** (`mentor`) | opponentSim · judgeSim | — | 2 |
| **Старший партнёр** | — | — | 1 DeepSeek V4 Pro |

**Максимум агентов за один прогон (все 4 модуля включены):**

```
контекст(1) + сегментатор(1)
+ Аудитор: redFlags(1) + collisions(1) + procKiller(1) + factCheck-экстрактор(1)   = 4
+ Стратег: heatmap(1) + контраргументы(до 6)                                       = 7
+ Драфтер(1)
+ Ментор: opponentSim(1) + judgeSim(1)                                             = 2
+ Старший партнёр(1, DeepSeek)
= 17 LLM-агентов  +  до 30 верификаторов статей (без LLM, внутри factCheck)
```

**Жёсткие лимиты:** контраргументов — **6** (`threats.slice(0,6)`),
red flags / коллизий / проц.дефектов — по 6–8, тепловая карта — 25 пунктов,
атак ментора — 5, вопросов судьи — 5.

**Порядок:** Аудитор ∥ Стратег → (Драфтер ∥ Ментор) → Старший партнёр.

# ГРУППА Г — СРАВНЕНИЕ РЕДАКЦИЙ (Semantic Legal Redlining)

Новый режим в IDE (вкладка **«Сравнение»** в pill «Чат · Агент · Сравнение»).
Юрист загружает две версии документа — система показывает **семантические**
изменения (не диффы букв), оценивает риски и выдаёт Executive Summary.

---

## Г.1 — Сравнение двух редакций (`/api/compare-documents`)

Файл: **`routes/compare.js`** (единственный режим, вынесенный в отдельный модуль —
фабрика, которой `server.js` передаёт хелперы; циркулярного require нет).

**Архитектура Map-Reduce** — четыре этапа:

| Этап | Что делает | Pinecone | LLM |
|---|---|---|---|
| **0. CONTEXT** | `extractDocumentContext` извлекает тип, отрасль и стороны договора (инжектируется в сегментатор и воркеры) | — | 1 |
| **1. ALIGN** | сегментация обоих документов с лимитом `COMPARE_SEG_LIMIT = 60` + эмбеддинги каждого пункта + жадное сопоставление по cosine similarity (порог **≥ 0.78**) | — (используются только эмбеддинги) | 2–4 (сегментаторы) |
| **2. MAP** | пары → 1 пара = 1 агент (`WORKER_BATCH_SIZE = 1`) → параллельные воркеры на `gemini-2.5-flash` (`temperature: 0.2`, `maxOutputTokens: 2048`) с учетом контекста документа дают вердикт: hasChanges / category / riskDetected / riskDescription | — | до **~100** воркеров (волнами по `WORKER_CONCURRENCY = 12`) |
| **3. REDUCE** | фильтр важных изменений → **компактная дельта** (НЕ полные тексты, чтобы уложиться в окно судьи 12K) → DeepSeek V4 Pro (с `reasoning_effort: 'medium'`) стримит Executive Summary | — | 1 DeepSeek (с откатом на Gemini) |

**Параметры:**

| Параметр | Значение | Где задано |
|---|---|---|
| Порог выравнивания cos-similarity | **0.78** | `ALIGN_THRESHOLD` |
| Размер батча воркера | **1 пара** | `WORKER_BATCH_SIZE` |
| Макс. пар в обработке | **100** | `MAX_PAIRS` |
| Макс. пунктов на документ | **60** | `COMPARE_SEG_LIMIT` (отдельный) |
| Мин. длина каждого документа | **50 симв.** | `MIN_DOC_LEN` |
| Лимит одновременных LLM-воркеров | **12** | `WORKER_CONCURRENCY` |
| Лимит одновременных эмбеддингов | **20** | `EMBED_CONCURRENCY` |
| Запрос для эмбеддинга пункта | до 1500 симв. | `EMBED_QUERY_MAX` |
| Текст пункта в промпте воркера | до 1200 симв. | `SEGMENT_TEXT_LIMIT` |
| Дельта для судьи | до 12000 симв. | `DELTA_CHARS_LIMIT` |
| temperature воркеров | **0.2** | inline в `runWorkerBatch` |
| temperature судьи | **0.25** | inline в вызове `streamDeepSeekResponse` |
| reasoning судьи (DeepSeek) | **medium** | inline в вызове `streamDeepSeekResponse` |
| Rate limit | **8 / мин** | отдельный `compareLimiter` |

**Категории изменений (выдаёт воркер для каждой пары):**

| Категория | Значение |
|---|---|
| `стилистика` | формулировки переписаны, юр. смысл не изменился |
| `существенное изменение` | права/обязанности/сроки/суммы реально изменены |
| `добавление` | пункт есть только в новой редакции |
| `удаление` | пункт был в старой, в новой отсутствует |

`riskDetected: true` ставится ТОЛЬКО когда новая редакция объективно хуже для
нашей стороны (выше ответственность, сокращены права, иная подсудность и т.п.).

**Максимум агентов за один прогон:**

```
1 экстрактор контекста
+ 2 сегментатора (старый ∥ новый)
+ до 200 эмбеддингов (100 + 100) — без LLM
+ до 100 воркеров-аудиторов (1 пара = 1 агент) — параллельно (волнами по 12)
+ 1 Старший партнёр (DeepSeek V4 Pro) — стрим финального резюме (reasoning: medium)
= до 104 LLM-агентов + 200 эмбеддингов
```

**SSE-события клиенту** (фронт `<CompareMode/>` уже умеет их рендерить):

| Событие | Когда |
|---|---|
| `{ step: { id, status, text } }` | для каждого этапа: segment / embed / align / map / judge |
| `{ protocolStatus }` | статусные строки для тоста |
| `{ compareReport: { total, matched, added, removed, risksCount, substantialCount, pairs[] } }` | один раз — ПОЛНЫЙ side-by-side отчёт ДО синтеза, фронт сразу рисует |
| `{ text }` | чанки Executive Summary от DeepSeek |
| `[DONE]` | конец |

**Промпты:** `WORKER_SYSTEM_PROMPT`, `JUDGE_SYSTEM_PROMPT_COMPARE` —
оба inline внутри `routes/compare.js`, у обоих явный запрет на упоминание
внутренней кухни системы.

**Фронтенд:** компонент `<CompareMode/>` в IDE — две текст-зоны (или
`.txt`-загрузка) → кнопка «Сравнить» → стриминговый степпер + Executive
Summary сверху + двухколоночный side-by-side viewer с цветной подсветкой
(🔴 риск · 🟡 сущ. изменение · 🟢 добавлено · ⚫ удалено) и кликом на пункт
для всплывающей карточки с описанием риска.

| Режим | Эмбеддинги | Pinecone (topK) | LLM Gemini | DeepSeek |

## 3. Справочник промптов

| Константа | Режим |
|---|---|
| `systemInstruction` | Болталка, Быстрый |
| `BASE_CONSULTANT_PROMPT` | Думающий (Простой + Глубокий) |
| `ACADEMIC_PROMPT_ADDON` | + при запросе курсовой / СРС / реферата |
| `L4_WARNING_ADDON` | + при просьбе составить иск / жалобу в суд |
| `AGENT_SYSTEM_PROMPT` | Агент-редактор |
| `JUDGE_SYSTEM_PROMPT` | Финал «Проверки документа» |
| `SENIOR_PARTNER_PROMPT` | Финал «Глубокого анализа PRO» |
| `WORKER_SYSTEM_PROMPT` (compare.js) | Воркер-аудитор пар в «Сравнении» |
| `JUDGE_SYSTEM_PROMPT_COMPARE` (compare.js) | Старший партнёр — Executive Summary в «Сравнении» |

Промпты воркеров (экстрактор, сегментатор, посегментный судья, Аудитор,
Стратег, Драфтер, Ментор) заданы inline внутри соответствующих функций.
